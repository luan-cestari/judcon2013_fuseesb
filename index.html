Apache Camel

Some Integration Patterns:

  • Content Based Router
  • Message Filter
  • Receipent List
  • Splitter
  • Aggregator
  • Resequencer
  • Throttler
Content Based Router=You can put a logic to choose a specific physical server Message Filter=avoid unintresting messages Receipent List=route to a dynamyc list Splitter=message has mutiple elementes, each have to be processed in a diff way Aggregator=combine the result of related messages Resequencer=put the messeger in order Throttler=ensure endpoint not get overloaded or exceed SLA

Real world Example in Java:

							
from("ftp://john@localhost/ftp?password=nhoj")
	.to("xslt:MyTransform.xslt")
	.to("activemq:queue:MyQueue")
read an XML file from an FTP server, process it using XSLT, and then send it to a JMS queue

Introduction to OSGi

Problems of Java EE

  • Lack of modularity
  • Service Tracking
  • Criteria-based service selection
  • Life cyle for Components
  • Mutiple versions of Components
  • Hot-swapping of Components
All java projects have the mod. problem, the Jigsaw project target to solve this (in Java 9) Jar hell and many dependecies not necessary

OSGi

Bundle / Component Oriented, Run Mutiple Applications, Single VM, Separate Classloader per bundle, Dynamic bundle/modules Specification cover Blueprint, Remote Service, JNDI, JTA, JDBC, Web Applications, etc The bundle/modules are components with specific version, that can be managed/changed in runtime Bundle=project/component + metadata OSGi container manager the bundle and the bundaries between bundles There is a Life Cycle to guaratee the coehesion and enforces sophisticated OSGi uses by default the MANIFEST.MF where should be specified the versions It remember maven, acctually the Karaf have a service that could create the Manifest from POM

Fuse Fabric

Why:

  • ServiceMix and ActiveMQ is complex to Configure
  • Require a lot of upfront knowledge
  • Different environment (private/public/hybrid cloud)
  • Location
    • Easy to go wrong

Fuse Fabric - Features

Support Hybrid deployments on premise, on cloud, on both

  • Endpoints can be relocated
  • Endpoints can be load balanced
  • Endpoints can be elastic
  • Endpoints can be highly available
